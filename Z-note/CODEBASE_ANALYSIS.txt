3. G2BULK — PRODUCT PURCHASE FLOW & API
================================================================================

Which games use G2Bulk?
-----------------------
Defined in routes/mlRoutes.js (lines 92-121):
  - hok          (Honor of Kings)       → provider = 'g2bulk'
  - pubgm        (PUBG Mobile)          → provider = 'g2bulk'
  - mlbb_special (ML Special SG/MY)     → provider = 'g2bulk'

API Configuration (from .env)
-----------------------------
  G2BULK_API_URL   — Base URL of the G2Bulk API
  G2BULK_API_KEY   — API key, sent as X-API-Key header + Authorization: Bearer

Client created in: controllers/mlController.js, function getG2BulkClientOrThrow()
(lines 182-195)

G2Bulk API Endpoints Called
---------------------------
  Function                  | G2Bulk API Endpoint              | Purpose
  ------------------------- | -------------------------------- | ---------------------------
  g2bulkFetchFields()       | POST /games/fields               | Get required input fields
  g2bulkCheckPlayerId()     | POST /games/checkPlayerId        | Validate player ID/server
  g2bulkCreateOrder()       | POST /games/{code}/order         | Create a purchase order
  shop() controller         | GET  /games/{code}/catalogue     | Fetch product catalogue

G2Bulk Purchase Flow (end-to-end)
---------------------------------
1. Customer visits shop page (e.g. /shop/hok)
   → mlController.shop() calls GET /games/hok/catalogue to list products,
     matched against locally-stored Product records for pricing.

2. Customer verifies player ID
   → Frontend calls POST /api/g2bulk/games/:code/check-player
   → mlController.g2bulkCheckPlayer()
   → calls POST /games/checkPlayerId on G2Bulk API.

3. Customer places order
   → Frontend calls POST /api/g2bulk/games/:code/order
   → mlController.g2bulkPlaceOrder() (line 1515):
     - Looks up G2BulkItem mapping to get fixed price from Product table
     - Re-verifies player ID via G2Bulk API
     - Re-fetches catalogue to get USD price
     - Applies fixed price (from admin) or dynamic pricing (USD × exchange rate)
     - Deducts wallet balance (with row-level locking)
     - Calls POST /games/{code}/order on G2Bulk API
     - Creates GamePurchaseTransaction record with provider: 'g2bulk'
     - Sends status email

4. Callback
   → G2Bulk calls POST /api/g2bulk/order-callback
   → mlController.g2bulkOrderCallback() (line 1791)
   → Updates status (processing → completed / fail)
   → Handles refunds on failure

Data Models
-----------
  G2BulkItem (models/G2BulkItem.js)
    — Maps local Product.id → g2bulkProductId (ID on G2Bulk's side)

  Product (models/Product.js)
    — Stores local pricing (price_mmk, price_thb) set by admin

  GamePurchaseTransaction (models/GamePurchaseTransaction.js)
    — Stores all orders with provider = 'g2bulk'

4. SMILE (SmileOne) — API DEFINITION & INTEGRATION
================================================================================

Which games use Smile?
----------------------
Defined in routes/mlRoutes.js (lines 97-116):
  - ml     (Mobile Legends Global)     → provider = 'smile'
  - mlphp  (Mobile Legends PH)         → provider = 'smile'
  - mcgg   (Magic Chess Go Go)         → provider = 'smile'

API Configuration (from .env)
-----------------------------
  ML_API_URL     = https://www.smile.one  (base URL)
  ML_API_EMAIL   — SmileOne account email
  ML_API_UID     — SmileOne account UID
  ML_API_KEY     — SmileOne API key (used for MD5 signature generation)
  ML_API_PRODUCT = mobilelegends  (default product slug)

Smile API Endpoints Called
--------------------------
Defined in controllers/mlController.js (lines 58-150):

  Function                  | Smile API Endpoint                           | Purpose
  ------------------------- | -------------------------------------------- | -------------------------
  smileOneGetRoleInfo()     | POST {ML_API_URL}/smilecoin/api/getrole      | Verify player account
                            | (or /ph/smilecoin/api/getrole for PH region) |
  smileOneCreateOrder()     | POST {ML_API_URL}/smilecoin/api/createorder  | Place diamond top-up order
                            | (or /ph/smilecoin/api/createorder for PH)    |

Both use MD5 double-hash signature generated by generateSign() (line 45).

Smile Purchase Flow
-------------------
1. Customer visits shop (e.g. /shop/ml)
   → Products loaded from local Product table (no external API call for catalogue).

2. Customer verifies player
   → Frontend calls POST /api/verify-user
   → mlController.verifyUser() (line 960)
   → calls smileOneGetRoleInfo() with appropriate productSlug and region.

3. Customer places order
   → Frontend calls POST /api/place-order
   → mlController.placeOrder() (line 1011):
     - Loads SmileSubItem records linked to chosen Product
       (or falls back to smileIDCombination field)
     - Deducts wallet balance (with row-level locking)
     - Loops through each SmileSubItem, calls smileOneCreateOrder() for each
     - Supports partial success — if some sub-orders fail, only successful
       portion is charged
     - Creates GamePurchaseTransaction with provider: 'smile'
     - Sends confirmation email

Data Models
-----------
  SmileSubItem (models/SmileSubItem.js)
    — Maps Product.id → one or more smileProductId values
    — Has region field: 'b' (global) or 'ph' (Philippines)

  Product (models/Product.js)
    — Also has legacy smileIDCombination field as fallback
